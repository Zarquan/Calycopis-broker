    Separate package - validation.

    Validation chain deals with the API package interfaces with UUID references.
    Validation chain calls context to lookup UUID references.
    Much like factory select(UUID)

    At the end, validation Result has a Builder.
    Builder creates an Entity.
        public AbstractDataResourceEntity build(final SimpleExecutionSessionEntity session);

    Builder has a link to the context, which has the lists of other results.
    Results have getEntity() to get the entity.
    Which links validation to engine.

    So we want a factory for ResultBeans that is pulled in from engine ?
    Validator could use Platform to get the EntityFactories ?
    Result.getEntity() is used to link data, storage, and volume mounts, during the build step.
    Can we move the build() part into engine ?

        AbstractDataResourceValidator.Result dataResult = new AbstractDataResourceValidator.ResultBean(
            Validator.ResultEnum.ACCEPTED,
            validated
            ){
            @Override
            public AbstractDataResourceEntity build(final SimpleExecutionSessionEntity session)
                {
                this.entity = IvoaDataResourceValidatorImpl.this.entityFactory.create(
                    session,
                    storage.getEntity(),
                    this
                    );
                return this.entity;
                }

    Engine provides a ResultBeanFactory
    Platform.getAbstractDataResourceValidatorResultFactory
    => new AbstractDataResourceValidator.ResultBean(
        ....
        ){
        @Override
        public AbstractDataResourceEntity build(final SimpleExecutionSessionEntity session)
            {}
        }

    I think we can get rid of the getEntity() method if we include a factory for finding them instead.
    Rather than
                this.entity = IvoaDataResourceValidatorImpl.this.entityFactory.create(
                    session,
                    storage.getEntity(),
                    this
                    );

                this.entity = IvoaDataResourceValidatorImpl.this.entityFactory.create(
                    session,
                    storageResolver.select(storage.getUuid()),
                    this
                    );


    OK, we can do this, but do we need to do it now ?
    Keep validation, entities, and business logic in engine for now.

    The target is to separate out the spring components.
    Separating the validation can come later.

    -- leave validation for later --

    * Validation stays in calycopis-broker-engine but uses calycopis-broker-api for the beans.
    * Inbound, calycopis-broker-spring deserialises requests into beans that implement the API.
    * Validation operates on the API bean from calycopis-broker-spring and factories from calycopis-broker-engine to create validated beans and then entities.
    * Outbound, calycopis-broker-engine provides entities that implement the API,
      and then calycopis-broker-spring provides a factory that serialises API beans into responses.

    We probably need to look at the build() stage of validation in more detail.
