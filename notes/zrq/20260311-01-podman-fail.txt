  Here are the steps needed to add a configurable exit code parameter:
  1. Update `entrypoint.sh` — Accept an optional second argument for the exit code. This means:
    • Keep $1 (seconds) as the required first argument.
    • Add $2 (exit code) as an optional second argument, defaulting to 0 if not provided.
    • Validate that $2 is a non-negative integer when supplied.
    • After the sleep completes, exit with the specified code instead of the implicit 0.
    • Update the usage message to reflect the new parameter.
  2. Update `Dockerfile` — No changes strictly needed, since the Dockerfile just copies and sets the
     entrypoint. However, if you wanted to document default behavior via CMD, you could add a CMD
     ["0", "0"] default (optional).
  3. Update `README.md` — Document the new parameter in the usage section, e.g.:


     docker run cantliei <seconds> [exit-code]

     Add an example showing a non-zero exit, such as docker run cantliei 5 1.
  4. Update `ivoa-execution.yaml` — Update the command template and description to reflect the second
     parameter, e.g.:


     command: "<seconds> <exit-code>"

  5. Rebuild and test — Verify the three main cases:
    • docker run cantliei 5 — waits 5 seconds, exits 0 (backward compatible).
    • docker run cantliei 5 1 — waits 5 seconds, exits 1.
    • docker run cantliei 5 abc — prints a validation error.
  6. Commit, push, and let CI publish — Once merged to main, the GitHub Actions workflow will build
     and push a new image to GHCR. After that, update the image.locations tag and digest in
     ivoa-execution.yaml to reference the newly published image.

  The key design decision is whether the exit code should be optional (defaulting to 0 for backward
  compatibility) or required. Making it optional is the safer choice since existing users won't need to
  change anything.
