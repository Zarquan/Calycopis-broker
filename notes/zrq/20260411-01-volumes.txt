    >     The suggested plan is sound, but the entity relationships need to be navigated carefully. Here's the data model chain
    >     visible in the test output and code:
    >     Request flow: A SimpleVolumeMount in the request references a data resource by name. The DockerDataStorageLinkerImpl sees
    >      the file:// URL on the data resource and creates a DockerBindMountStorageEntity with the host path. The volume mount
    >     entity (SimpleVolumeMountEntity) links the compute resource to either the data resource or the storage resource.
    >     Entity relationships on `AbstractVolumeMountEntity`:
    >     • getComputeResource() → the compute resource (always set)
    >     • getDataResource() → the data resource (nullable)
    >     • getStorageResource() → the storage resource (nullable)
    >   
    >     The key navigation path for bind mounts is:
    >   
    >     ComputeResource
    >       └─ volumeMounts[] (SimpleVolumeMountEntity)
    >            ├─ getPath()        → container mount path (e.g. "/input")
    >            ├─ getMode()        → READONLY / READWRITE
    >            └─ getDataResource() → AbstractDataResourceEntity
    >                 └─ getStorage() → AbstractStorageResourceEntity
    >                      └─ if instanceof DockerBindMountStorage
    >                           └─ getMountPath() → host path (e.g. "/home/Zarquan/temp/random.txt")
    >   
    >     So the plan should iterate this.getVolumeMounts(), and for each volume mount, follow the chain through the data resource
    >     to its storage, check if the storage is a DockerBindMountStorage, and if so build a Bind entry from

    >     The key insight is that instanceof fails on AbstractStorageResourceEntity$HibernateProxy because the proxy wraps the
    >     abstract base class. But when you call a method on a Hibernate proxy, the proxy delegates to the real underlying entity.
    >     So storage.link(linker) will dispatch to DockerBindMountStorageEntity.link() or DockerVolumeMountStorageEntity.link()
    >     correctly — even though the calling code only sees AbstractStorageResource. The polymorphic method dispatch bypasses the
    >     proxy issue entirely.

    >     • The AbstractStorageLinker / DockerStorageLinker split keeps Docker API dependencies out of the abstract layer. A mock

    >       construct the correct Bind without any external casting or unproxying.
