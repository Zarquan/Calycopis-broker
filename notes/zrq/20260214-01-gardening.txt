  Here is my analysis of the IVOA ExecutionBroker 1.0.3 OpenAPI specification.

  ────────────────────────────────────────

  Overview

  • OpenAPI version: 3.1.0
  • Title: IVOA ExecutionBroker
  • API version: 1.0
  • Author: Dave Morris (openapi.dev@dm92.uk)
  • License: CC-BY-SA-4.0
  • Server: http://localhost:8080 (development only)

  This API defines an execution brokering service -- a system where clients submit execution requests (describing what to run and what resources are needed), receive back a set of offers
  from the broker, and then manage execution sessions through a lifecycle.

  ────────────────────────────────────────

  Endpoints (3 paths, 4 operations)


  ┌────────┬───────────────────┬──────────────────────┬─────────────────────────────────────────────┐
  │ Method │ Path              │ OperationId          │ Description                                 │
  ├────────┼───────────────────┼──────────────────────┼─────────────────────────────────────────────┤
  │ POST   │ /offersets        │ OfferSetPost         │ Submit an execution request; receive offers │
  │ GET    │ /offersets/{uuid} │ OfferSetGet          │ Retrieve a previously created offer set     │
  │ GET    │ /sessions/{uuid}  │ ExecutionSessionGet  │ Retrieve an execution session               │
  │ POST   │ /sessions/{uuid}  │ ExecutionSessionPost │ Update an execution session                 │
  └────────┴───────────────────┴──────────────────────┴─────────────────────────────────────────────┘

  All endpoints support JSON, XML, and YAML content types for both request and response bodies.

  Notable response patterns

  • POST /offersets can return either:
    • 303 (redirect) with a Location header pointing to /offersets/{uuid} (async pattern), or
    • 200 with the OfferSetResponse inline (sync pattern)
  • The session POST endpoint accepts AbstractUpdate objects for modifying session state.

  ────────────────────────────────────────

  Schema Architecture (55 schemas)

  The spec uses a heavily polymorphic, inheritance-based design with allOf composition and discriminator mappings. The kind field (a URI) serves as the discriminator across most abstract
  types.

  Core Domain Model

  1. Request/Response flow:
  • OfferSetRequest = ExecutionRequestComponents + RequestedScheduleBlock
  • OfferSetResponse = result (YES/NO) + array of AbstractExecutionSession offers

  2. Execution Components (ExecutionRequestComponents / SimpleExecutionComponents):

  ┌────────────┬─────────────────────────────┬─────────────────────────┐
  │ Property   │ Type                        │ Purpose                 │
  ├────────────┼─────────────────────────────┼─────────────────────────┤
  │ executable │ AbstractExecutable          │ What to run             │
  │ compute    │ AbstractComputeResource     │ CPU/memory requirements │
  │ storage    │ AbstractStorageResourceList │ Persistent storage      │
  │ volumes    │ AbstractVolumeMountList     │ Filesystem mounts       │
  │ data       │ AbstractDataResourceList    │ Data inputs             │
  └────────────┴─────────────────────────────┴─────────────────────────┘

  3. Executable types (discriminated by kind URI):
  • DockerContainer -- OCI/Docker containers with image spec, entrypoint, environment, network ports, privileged mode
  • SingularityContainer -- Singularity containers (URL-based location)
  • JupyterNotebook -- Jupyter notebooks (URL-based location)

  4. Resource types:
  • Compute: SimpleComputeResource with min/max cores and memory (GiB)
  • Storage: SimpleStorageResource with min/max size (GiB)
  • Volumes: SimpleVolumeMount with path, mode (READONLY/READWRITE), cardinality (SINGLE/CONTAINER)
  • Data: 5 concrete types:
    • SimpleDataResource -- plain URL download
    • S3DataResource -- S3 bucket/object
    • RucioDataResource -- Rucio scientific data management
    • IvoaDataResource -- IVOA registry resources (with ObsCore and DataLink metadata)
    • SkaoDataResource -- extends IvoaDataResource with SKA Observatory data lake metadata

  5. Session types:
  • SimpleExecutionSession -- phase, expiry, components, connectors
  • ScheduledExecutionSession -- extends Simple with a schedule (preparing/available/releasing intervals)

  6. Session phases (13 states):
  INITIAL -> OFFERED -> ACCEPTED/REJECTED/EXPIRED -> WAITING -> PREPARING -> AVAILABLE -> RUNNING -> RELEASING -> COMPLETED/FAILED/CANCELLED
  7. Update mechanism (discriminated by kind):
  • StringValueUpdate / EnumValueUpdate / IntegerValueUpdate / IntegerDeltaUpdate
  • Each targets a path within the session to modify

  8. Options mechanism (mirrors updates, offered by server):
  • StringValueOption / EnumValueOption / IntegerValueOption / IntegerDeltaOption
  • Advertises what the client can change, with constraints (patterns, enums, min/max)


  ────────────────────────────────────────

  Design Patterns & Observations

  Strengths:
  • Clean separation of concerns -- executables, compute, storage, data, and volumes are all independently polymorphic
  • The options/update mechanism is elegant: the server describes what is negotiable (options), and the client responds with specific changes (updates), both using path-based targeting
  • Good use of ISO 8601 patterns for scheduling (instants, durations, intervals)
  • Multi-format support (JSON/XML/YAML) throughout
  • The SkaoDataResource -> IvoaDataResource hierarchy shows good domain specialization

  Potential concerns:
  • ExecutionRequestComponents and SimpleExecutionComponents are structurally identical but separate schemas -- this could be consolidated
  • LifecycleStartDurationInstant/LifecycleStartDurationInterval and ScheduleStartDurationInstant/ScheduleStartDurationInterval are duplicated pairs with the same structure
  • The LifecyclePhase enum and SimpleExecutionSessionPhase enum overlap significantly but are separate types with slightly different values (LifecyclePhase has INITIALIZING, session has
     INITIAL/OFFERED/ACCEPTED/REJECTED/EXPIRED)
  • Several description fields contain trailing whitespace/newlines (minor formatting issue)
  • POST /sessions/{uuid} response description is a placeholder: "Required description ..."
  • The ISO8601Instant regex has a possessive quantifier (?>...) which may not be supported in all regex engines
  • No error response schemas are defined (no 4xx/5xx responses specified)
  • No authentication/security schemes are defined
  • The DockerContainer.environment points to NameValueMap (additionalProperties map) which is good, but volumes in SimpleComputeResource is just array[string] (references by name/UUID?)
     -- the linkage mechanism between volumes and compute isn't well documented
  • The JupyterNotebook location field has a TODO comment indicating it's still under design


  ────────────────────────────────────────

  Summary

  This is a well-structured IVOA standard API for brokering scientific computation execution. The core workflow is: request -> offer set -> accept offer -> session lifecycle management. It
   supports containerized workloads (Docker, Singularity) and Jupyter notebooks, with rich support for scientific data infrastructure (S3, Rucio, IVOA registries, SKAO data lake). The
  polymorphic design allows extensibility via new kind URIs without breaking the schema.

--END--

  Clean build, no errors. Here's a summary of the refactoring for issue #340:

  ────────────────────────────────────────

  Changes across 24 files:
  1. `Validator.validate()` now returns `void` (Validator.java)
  • The return type changed from Validator.Result<ObjectType, EntityType> to void. Validators add their results directly to the context, so the return value was redundant.

  2. New `dispatched()` flag on context (OfferSetRequestParserContext.java, OfferSetRequestParserContextImpl.java)
  • Added dispatched(boolean) / dispatched() methods to track whether a validator recognised and handled the object. This replaces the CONTINUE/ACCEPTED/FAILED return-value dispatch
    mechanism.

  3. Factory loop refactored (ValidatorFactoryBaseImpl.java)
  • The loop now resets context.dispatched(false) before each validator call and checks context.dispatched() after. If dispatched, the factory returns immediately; if no validator
    handles the object, unknown() is called.

  4. All 5 abstract validator interfaces changed validate() to return void.
  5. All 5 factory impls had their "Nasty class cast :-(" overrides removed entirely. These existed solely to cast Validator.Result<O,E> to the domain-specific Result subtype, which is
  unnecessary with void.
  6. All 9 concrete validators updated:
  • validate() methods return void
  • The CONTINUE case (type mismatch) now simply returns without doing anything
  • The ACCEPTED and FAILED cases call context.dispatched(true) to signal they handled the object

  7. `AbstractDataResourceValidatorImpl.storageCheck()` refactored to look up the storage result from the context after the void validate call, instead of using the return value.
  8. `OfferSetRequestParserImpl.validate()` significantly simplified -- all the result.getEnum() == FAILED checks and context.valid(false) calls are removed since validators now handle
  this internally. This also fixed a pre-existing bug where computeResult.getEnum() was checked instead of executableResult.getEnum() on line 246.
