  Build succeeds cleanly. Here's a summary of all the changes:


  1. `LifecycleComponent.java` — new interface method

  Added getMonitorAction(ComponentProcessingRequest) alongside the existing getPrepareAction, getReleaseAction, etc.


  2. `LifecycleComponentEntity.java` — default implementation

  Added a default getMonitorAction() returning NO_ACTION, so existing components (mock, executables, storage, data) are unaffected.


  3. `MonitorComponentRequestEntity.java` — new file

  Follows the exact same pattern as PrepareComponentRequestEntity:
  • `preProcess`: Resolves the component via the platform, checks the phase is AVAILABLE or RUNNING, and calls
    component.getMonitorAction(this).
  • `postProcess`: Based on the action's getNextPhase():
    • RUNNING → sets phase to RUNNING, reschedules itself with a 10-second poll interval.
    • COMPLETED → sets phase to COMPLETED, notifies the session via createMonitorSessionRequest, marks itself done.
    • FAILED → fails the component and session.



  4. `ComponentProcessingRequestFactory` + `Impl` — new factory method

  Added createMonitorComponentRequest(component) to create and persist a MonitorComponentRequestEntity.


  5. `PrepareComponentRequestEntity.java` — wiring

  When a component's prepare action finishes with AVAILABLE, the post-process now also schedules a createMonitorComponentRequest for that
  component, kicking off the monitoring loop.


  6. `DockerSimpleComputeResourceEntity.java` — monitor implementation

  • Extracted buildDockerClient() as a static helper (reusable by both prepare and monitor).
  • getMonitorAction() eagerly captures the dockerContainerId, then returns a ProcessingAction that:
    • Calls dockerClient.inspectContainerCmd(containerId).exec() to get the container state.
    • If state.getRunning() is true → nextPhase = RUNNING.
    • If not running and exitCode == 0 → nextPhase = COMPLETED.
    • If not running and exitCode != 0 → nextPhase = FAILED.

  This creates a polling loop: once the container is started (AVAILABLE), the monitor checks every 10 seconds, transitions through RUNNING, and
   eventually reaches COMPLETED or FAILED when the container exits.
